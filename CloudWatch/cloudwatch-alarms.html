CloudWatch Alarms

Alerting & Automated Actions

Alarms Overview

CloudWatch alarms watch a single metric over a time period and perform one or more actions based on the value of the metric relative to a given threshold over a number of time periods.

Monitor

Watch metric values

Alert

Notify when thresholds breach

Act

Trigger automated responses

Alarm States

OK

The metric is within the defined threshold. The alarm is not triggered.

ALARM

The metric has breached the threshold. Actions are triggered.

INSUFFICIENT_DATA

Not enough data to determine the alarm state.

Alarm Components

Metric Configuration

Required Parameters

  • • Metric Name: The metric to monitor
  • • Namespace: Metric namespace
  • • Dimensions: Metric dimensions
  • • Statistic: Average, Sum, Min, Max, etc.

Threshold Settings

  • • Comparison Operator: >, >=, <, <=
  • • Threshold Value: Numeric threshold
  • • Period: Evaluation period (60s, 300s, etc.)
  • • Evaluation Periods: Number of periods

Advanced Configuration

Datapoints to Alarm

Specify how many datapoints within the evaluation period must be breaching to trigger the alarm.

Example: 2 out of 3 datapoints must breach

Missing Data Treatment

  • • notBreaching: Treat as not breaching
  • • breaching: Treat as breaching
  • • ignore: Ignore missing data
  • • missing: Evaluate as missing

Alarm Actions

SNS Notifications

Send notifications to SNS topics when alarm state changes. SNS can then deliver notifications via email, SMS, HTTP endpoints, or other AWS services.

Notification Types

• Email notifications
• SMS messages
• HTTP/HTTPS endpoints
• Lambda functions
• SQS queues
• Mobile push notifications

Auto Scaling Actions

Trigger Auto Scaling policies to automatically adjust the number of EC2 instances based on demand.

Scale Out (Add Instances)

  • • High CPU utilization
  • • High network traffic
  • • Queue depth increase
  • • Custom application metrics

Scale In (Remove Instances)

  • • Low CPU utilization
  • • Reduced network traffic
  • • Queue depth decrease
  • • Cost optimization triggers

EC2 Actions

Perform actions on EC2 instances when alarms trigger. Useful for automated recovery and maintenance.

Stop Instance

Stop EC2 instance

Terminate Instance

Terminate EC2 instance

Reboot Instance

Reboot EC2 instance

Composite Alarms

Composite alarms combine multiple alarms using Boolean logic (AND, OR, NOT) to reduce alarm noise and create more sophisticated alerting rules.

Benefits

  • • Reduce false positives
  • • Complex alerting logic
  • • Centralized alarm management
  • • Hierarchical alarm structure

Use Cases

  • • Application health monitoring
  • • Multi-tier architecture alerts
  • • Service dependency tracking
  • • Business process monitoring

Example Rule

(CPUAlarm OR MemoryAlarm) AND NOT MaintenanceMode

Alarm Best Practices

Threshold Selection

Set thresholds based on historical data and business requirements. Avoid setting thresholds too sensitive to prevent false alarms.

Evaluation Periods

Use multiple evaluation periods to avoid triggering on temporary spikes. Consider using "M out of N" datapoints for more reliable alerting.

Action Grouping

Group related alarms and use composite alarms to reduce notification noise and create meaningful alert hierarchies.

Testing & Validation

Regularly test alarm configurations and validate that actions work as expected. Use alarm history for troubleshooting.

Key Takeaways

  • Alarms monitor metrics and trigger actions when thresholds are breached
  • Three alarm states: OK, ALARM, and INSUFFICIENT_DATA
  • Actions include SNS notifications, Auto Scaling, and EC2 instance actions
  • Composite alarms enable complex alerting logic and reduce false positives
  • Proper threshold and evaluation period configuration is crucial for effective alerting
