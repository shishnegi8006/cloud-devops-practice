IAM Policies Deep Dive

Policy Types, Evaluation Logic & Best Practices

Policy Types

AWS IAM supports multiple policy types that work together to control access to resources.

Identity-Based Policies

  • • Attached to users, groups, or roles
  • • Define what actions the identity can perform
  • • Managed or inline policies
  • • Most common policy type

Resource-Based Policies

  • • Attached directly to resources
  • • Define who can access the resource
  • • S3 bucket policies, KMS key policies
  • • Support cross-account access

Permission Boundaries

  • • Maximum permissions for an entity
  • • Used with identity-based policies
  • • Delegated administration use case
  • • Advanced feature for large organizations

Service Control Policies (SCPs)

  • • AWS Organizations feature
  • • Maximum permissions for accounts
  • • Guardrails for organizational units
  • • Cannot grant permissions, only restrict

Policy Evaluation Logic

AWS evaluates policies in a specific order to determine if a request is allowed or denied.

Policy Evaluation Flow
1
Explicit Deny
Check for explicit deny in any policy
DENY
2
Service Control Policies
Check SCPs for account/OU restrictions
FILTER
3
Resource-Based Policies
Check if resource policy allows
ALLOW
4
Identity-Based Policies
Check identity permissions
ALLOW
5
Permission Boundaries
Check maximum permissions
LIMIT

Default Behavior

By default, all requests are denied. An explicit allow is required from an applicable policy.

Policy Structure

IAM policies are JSON documents with specific elements that define permissions.

Element Required Description
Version Yes Policy language version
Statement Yes Array of permission statements
Effect Yes Allow or Deny
Action Yes* API actions to allow/deny
Resource Yes* Resources the actions apply to
Principal No Who the policy applies to
Condition No Circumstances for policy application
// Basic policy structure
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::bucket/*"
    }
  ]
}

Variables & Conditions

Use policy variables and conditions to create dynamic, context-aware policies.

Policy Variables

  • • ${aws:username} - IAM user name
  • • ${aws:userid} - Unique user ID
  • • ${aws:PrincipalTag/TagKey} - Principal tags
  • • ${aws:RequestedRegion} - Requested region

Date/Time Conditions

  • • DateGreaterThan - After specific date
  • • DateLessThan - Before specific date
  • • DateEquals - On specific date

IP Address Conditions

  • • IpAddress - Allow from specific IPs
  • • NotIpAddress - Deny from specific IPs
  • • aws:VpcSourceIp - VPC source IP
// Condition example
"Condition": {
  "IpAddress": {
    "aws:SourceIp": "203.0.113.0/24"
  },
  "DateGreaterThan": {
    "aws:CurrentTime": "2024-01-01T00:00:00Z"
  }
}

Managed vs Inline

AWS Managed Policies

  • • Created and maintained by AWS
  • • Automatically updated for new services
  • • Cannot be modified
  • • Examples: PowerUserAccess, ReadOnlyAccess

Customer Managed Policies

  • • Created and maintained by you
  • • Reusable across multiple entities
  • • Version control and rollback
  • • Up to 6,144 characters

Inline Policies

  • • Directly embedded in single entity
  • • One-to-one relationship
  • • Deleted when entity is deleted
  • • Use for unique, specific permissions

Best Practice

Use managed policies for common permissions, inline policies for exceptions or unique requirements.

Best Practices

Principle of Least Privilege

  • • Grant minimum permissions required
  • • Start with deny all, add permissions as needed
  • • Regular access reviews and cleanup
  • • Use IAM Access Analyzer

Policy Organization

  • • Use groups for common permissions
  • • Consistent naming conventions
  • • Document policy purposes
  • • Version control for custom policies

Monitoring & Auditing

  • • Enable CloudTrail for API logging
  • • Use IAM credential reports
  • • Monitor unused permissions
  • • Set up access pattern alerts

Security Considerations

  • • Avoid wildcards in production
  • • Use conditions to restrict access
  • • Implement MFA for sensitive operations
  • • Regular policy validation and testing
