IAM Identity Federation

SAML, OIDC, and External Identity Providers

Federation Overview

Identity federation allows users to access AWS resources using credentials from external identity providers.

SAML 2.0 Federation

  • • Enterprise identity providers (AD FS, Okta)
  • • XML-based authentication protocol
  • • Single sign-on (SSO) capabilities
  • • Supports both console and API access

OpenID Connect (OIDC)

  • • Modern identity layer on OAuth 2.0
  • • JSON Web Tokens (JWT)
  • • Web and mobile applications
  • • GitHub Actions, GitLab CI/CD

Web Identity Federation

  • • Social identity providers
  • • Amazon, Facebook, Google, Apple
  • • Mobile and web applications
  • • Direct STS integration

Amazon Cognito

  • • Managed identity service
  • • User pools and identity pools
  • • Multiple identity providers
  • • Simplified mobile/web integration

SAML 2.0 Federation

Security Assertion Markup Language (SAML) enables enterprise SSO integration with AWS.

SAML Federation Flow
1
User Authentication
User authenticates with IdP
2
SAML Assertion
IdP generates SAML response
3
AssumeRoleWithSAML
STS validates and issues tokens
4
AWS Access
User accesses AWS resources

Setup Requirements

  • • Create SAML identity provider in IAM
  • • Upload IdP metadata document
  • • Create IAM role for SAML users
  • • Configure trust relationship
# Create SAML provider
aws iam create-saml-provider \
  --saml-metadata-document file://metadata.xml \
  --name MyCompanySAMLProvider

OpenID Connect (OIDC)

Modern identity protocol built on OAuth 2.0, ideal for CI/CD pipelines and web applications.

  • JWT Tokens: JSON Web Tokens for identity claims
  • Thumbprint Validation: Certificate fingerprint verification
  • Audience Restriction: Limit token usage to specific applications
  • Short-lived Tokens: Enhanced security with token expiration
GitHub Actions
GitLab CI/CD
Google
Custom OIDC

CI/CD Use Cases

  • • GitHub Actions deploying to AWS
  • • GitLab pipelines accessing resources
  • • No long-term credentials in code
  • • Temporary access per workflow run
# Create OIDC provider
aws iam create-open-id-connect-provider \
  --url https://token.actions.githubusercontent.com \
  --client-id-list sts.amazonaws.com \
  --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1

Web Identity Federation

Enable users to sign in with social identity providers and access AWS resources.

Provider Use Case Token Type
Amazon Login with Amazon OAuth 2.0
Facebook Social login OAuth 2.0
Google Google Sign-In OAuth 2.0
Apple Sign in with Apple OAuth 2.0

Mobile Applications

  • • Direct integration with AWS SDK
  • • AssumeRoleWithWebIdentity API
  • • No server-side authentication needed
  • • Temporary credentials for mobile apps

Considerations

  • • Token validation by AWS STS
  • • Provider-specific configuration
  • • Rate limiting and quotas
  • • Consider Cognito for complex scenarios
// JavaScript SDK example
AWS.config.credentials = new AWS.WebIdentityCredentials({
  RoleArn: 'arn:aws:iam::123456789012:role/WebIdRole',
  ProviderId: 'graph.facebook.com',
  WebIdentityToken: facebookToken
});

Amazon Cognito

Managed service that provides authentication, authorization, and user management for web and mobile apps.

User Pools

  • • User directory service
  • • Sign-up and sign-in functionality
  • • MFA and password policies
  • • Custom authentication flows

Identity Pools

  • • Federated identities
  • • AWS credentials for authenticated users
  • • Support for multiple identity providers
  • • Guest (unauthenticated) access

Advanced Features

  • • Lambda triggers for custom logic
  • • Advanced security features
  • • Analytics and user insights
  • • SDK integration for multiple platforms

When to Use Cognito

  • • Complex user management requirements
  • • Multiple identity providers
  • • Mobile and web applications
  • • Need for user attributes and groups

Federation Best Practices

Security

  • • Use conditions to restrict access
  • • Validate token audiences and issuers
  • • Implement proper certificate validation
  • • Monitor federation usage patterns

Token Management

  • • Use short-lived tokens when possible
  • • Implement token refresh mechanisms
  • • Handle token expiration gracefully
  • • Cache tokens appropriately

Monitoring

  • • Enable CloudTrail for STS API calls
  • • Monitor failed authentication attempts
  • • Set up alerts for unusual patterns
  • • Regular access reviews and audits

Common Pitfalls

  • • Overly permissive role policies
  • • Insufficient token validation
  • • Missing audience restrictions
  • • Inadequate error handling
