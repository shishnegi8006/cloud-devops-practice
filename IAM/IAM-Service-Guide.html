IAM Service Guide

Complete reference with features, options, and real-world use cases

Core IAM Features

Users

Individual identities with unique credentials for accessing AWS resources.

Use Case:

Create separate accounts for each developer on your team

Best Practice: Enable MFA for all users

Groups

Collections of users that share the same permissions.

Use Case:

Create "Developers" group with S3 read access, "Admins" group with full access

Best Practice: Assign permissions to groups, not individual users

Roles

Temporary permissions that can be assumed by users, applications, or services.

Use Case:

Allow EC2 instances to access S3 buckets without storing credentials

Best Practice: Use roles instead of access keys for applications

Policies

JSON documents that define permissions (what actions are allowed or denied).

Use Case:

Allow read access to specific S3 bucket but deny delete operations

Best Practice: Use AWS managed policies when possible

Multi-Factor Authentication

Additional security layer requiring a second form of authentication.

Use Case:

Protect admin accounts with Google Authenticator or hardware tokens

Best Practice: Mandatory for root account and privileged users

Access Keys

Programmatic access credentials for API calls and CLI usage.

Use Case:

Allow applications to upload files to S3 or send emails via SES

Best Practice: Rotate keys regularly and use least privilege

Advanced IAM Features

Cross-Account Access

Allow users from one AWS account to access resources in another account.

Real-World Example:

Development team accesses production S3 buckets for troubleshooting

Identity Federation

Allow users to sign in using external identity providers (Google, Microsoft, etc.).

Real-World Example:

Employees use company Active Directory to access AWS console

Permission Boundaries

Set maximum permissions that a user or role can have.

Real-World Example:

Prevent developers from creating expensive resources like large EC2 instances

Access Analyzer

Identify resources shared with external entities and unused access.

Real-World Example:

Find S3 buckets accidentally made public or unused IAM roles

Common IAM Use Cases

Web Application Security

Create IAM role for EC2 instances to access RDS database

Use IAM policies to restrict S3 bucket access to specific IP ranges

Enable MFA for admin users managing production resources

Team Management

Create groups for different teams (Developers, QA, DevOps)

Assign environment-specific permissions (dev, staging, prod)

Use temporary credentials for contractors and external users
