RDS Parameter Groups

Database Engine Configuration & Tuning

What are Parameter Groups?

Parameter groups act as containers for engine configuration values that are applied to one or more DB instances. They provide a way to manage database engine settings and optimize performance for specific workloads.

Key Concepts: Engine-specific parameters, default and custom groups, static vs dynamic parameters, and parameter inheritance.

Parameter Group Types

DB Parameter Groups

For RDS Instances
  • • Single instance: Applied to individual RDS instances
  • • Engine-specific: Different parameters per database engine
  • • Version-specific: Parameters vary by engine version
  • • Default groups: AWS-provided default configurations
Supported Engines
  • • MySQL: my.cnf configuration parameters
  • • PostgreSQL: postgresql.conf parameters
  • • Oracle: Oracle initialization parameters
  • • SQL Server: SQL Server configuration options
  • • MariaDB: MariaDB-specific parameters

DB Cluster Parameter Groups

For Aurora Clusters
  • • Cluster-wide: Applied to entire Aurora cluster
  • • All instances: Affects all cluster members
  • • Aurora-specific: Aurora-optimized parameters
  • • Global settings: Cluster-level configuration
Aurora Engines
  • • Aurora MySQL: MySQL-compatible parameters
  • • Aurora PostgreSQL: PostgreSQL-compatible parameters
  • • Enhanced features: Aurora-specific optimizations
  • • Performance tuning: Cloud-native optimizations

Parameter Types & Behavior

Static Parameters

  • • Restart required: Changes need DB instance restart
  • • Persistent: Values persist across restarts
  • • System-level: Affect core database behavior
  • • Examples: innodb_buffer_pool_size, shared_preload_libraries
  • • Planning: Schedule maintenance windows for changes

Dynamic Parameters

  • • Immediate effect: Applied without restart
  • • Runtime changes: Modify during operation
  • • Session-level: Can affect individual sessions
  • • Examples: max_connections, query_cache_size
  • • Flexibility: Real-time performance tuning

System Parameters

  • • Read-only: Cannot be modified by users
  • • AWS-managed: Controlled by AWS infrastructure
  • • Instance-specific: Based on instance class
  • • Examples: innodb_buffer_pool_instances
  • • Optimization: AWS-optimized for performance

Common Parameters by Engine

MySQL Parameters

Performance Parameters
  • • innodb_buffer_pool_size: InnoDB buffer pool memory
  • • max_connections: Maximum concurrent connections
  • • query_cache_size: Query result cache size
  • • innodb_log_file_size: Transaction log file size
  • • tmp_table_size: Temporary table memory limit
Security Parameters
  • • ssl_ca: SSL certificate authority
  • • require_secure_transport: Force SSL connections
  • • log_bin_trust_function_creators: Binary logging security
  • • local_infile: Local file loading permissions

PostgreSQL Parameters

Memory Parameters
  • • shared_buffers: Shared memory buffer size
  • • work_mem: Memory for query operations
  • • maintenance_work_mem: Memory for maintenance operations
  • • effective_cache_size: OS cache size estimate
  • • max_connections: Maximum concurrent connections
Logging Parameters
  • • log_statement: SQL statement logging level
  • • log_min_duration_statement: Slow query threshold
  • • shared_preload_libraries: Preloaded extensions
  • • log_checkpoints: Checkpoint logging

Performance Tuning Examples

MySQL Optimization

Memory Optimization
  • • innodb_buffer_pool_size: 70-80% of available RAM
  • • query_cache_size: 32MB-256MB for read-heavy workloads
  • • tmp_table_size: 64MB-512MB based on temp table usage
  • • sort_buffer_size: 2MB-16MB for sorting operations
Connection Tuning
  • • max_connections: Based on application concurrency
  • • connect_timeout: 10-30 seconds for connection timeout
  • • wait_timeout: 300-3600 seconds for idle connections
  • • interactive_timeout: Match application behavior

PostgreSQL Optimization

Memory Configuration
  • • shared_buffers: 25% of RAM for dedicated DB server
  • • work_mem: 4MB-64MB per query operation
  • • maintenance_work_mem: 256MB-2GB for maintenance
  • • effective_cache_size: 75% of total system memory
Query Optimization
  • • random_page_cost: 1.1-2.0 for SSD storage
  • • seq_page_cost: 1.0 baseline for sequential reads
  • • default_statistics_target: 100-1000 for query planning
  • • checkpoint_completion_target: 0.7-0.9 for smooth I/O

SAA-C03 Exam Key Points

  • • Parameter Types: Static (restart required) vs Dynamic (immediate effect)
  • • Custom Groups: Cannot modify default parameter groups, must create custom
  • • Engine Specific: Parameter groups are engine and version specific
  • • Performance Tuning: Buffer pool size, connections, memory allocation
  • • Security: SSL enforcement, logging, access control parameters
  • • Monitoring: Use CloudWatch and Performance Insights to track impact
