Database Engines

RDS supports multiple database engines, each optimized for different use cases and workloads.

Engine Version Use Case
MySQL 8.0, 5.7 Web applications, OLTP
PostgreSQL 14, 13, 12 Complex queries, JSON
MariaDB 10.6, 10.5 MySQL alternative
Oracle 19c, 12c Enterprise applications
SQL Server 2019, 2017 Microsoft ecosystem
Aurora MySQL/PostgreSQL Cloud-native, high performance

Engine Selection Criteria

  • Application compatibility and requirements
  • Performance and scalability needs
  • Licensing costs and budget constraints
  • Feature requirements (JSON, spatial data)

Instance Classes

RDS instance classes are optimized for different workload patterns and performance requirements.

General Purpose (db.t3, db.m5)

  • • Balanced CPU, memory, and network
  • • Burstable performance (T3)
  • • Cost-effective for variable workloads
  • • 2-96 vCPUs, 1-384 GiB memory

Memory Optimized (db.r5, db.x1e)

  • • High memory-to-vCPU ratio
  • • In-memory databases and caching
  • • Real-time analytics workloads
  • • Up to 768 GiB memory per vCPU

Burstable Performance (db.t3, db.t2)

  • • CPU credits for burst capability
  • • Development and testing environments
  • • Low to moderate baseline performance
  • • Cost-effective for intermittent workloads
# Create RDS instance
aws rds create-db-instance \
  --db-instance-identifier mydb \
  --db-instance-class db.t3.micro \
  --engine mysql \
  --master-username admin \
  --master-user-password mypassword \
  --allocated-storage 20

Storage Types

RDS offers different storage types optimized for various performance and cost requirements.

General Purpose SSD (gp3)

  • • Latest generation, cost-effective
  • • 3,000 IOPS baseline, up to 16,000 IOPS
  • • 125 MB/s baseline, up to 1,000 MB/s
  • • Independent IOPS and throughput scaling

General Purpose SSD (gp2)

  • • 3 IOPS per GB, up to 16,000 IOPS
  • • Burst capability up to 3,000 IOPS
  • • Good for most workloads
  • • 20 GiB to 64 TiB storage

Provisioned IOPS SSD (io1/io2)

  • • Up to 64,000 IOPS (io2)
  • • Consistent, high IOPS performance
  • • I/O intensive applications
  • • 99.999% durability (io2)

Magnetic Storage

  • • Legacy storage type
  • • Lower cost, lower performance
  • • Not recommended for new deployments
  • • 10 GiB to 3 TiB storage

Multi-AZ Deployments

Multi-AZ provides high availability and automatic failover for RDS instances.

  • Synchronous Replication: Data replicated to standby in different AZ
  • Automatic Failover: 1-2 minutes failover time
  • No Performance Impact: Standby not accessible for reads
  • Maintenance Benefits: Zero-downtime maintenance

Benefits

  • • Enhanced availability and durability
  • • Protection against AZ failures
  • • Automatic database failover
  • • No manual intervention required

Considerations

  • • Additional cost (approximately 2x)
  • • Standby replica not readable
  • • Cross-AZ data transfer charges
  • • Use Read Replicas for read scaling
# Enable Multi-AZ
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --multi-az \
  --apply-immediately

Read Replicas

Read replicas provide read scaling and can be used for reporting and analytics workloads.

Scaling Capabilities

  • • Up to 5 read replicas per source DB
  • • Cross-region read replicas supported
  • • Asynchronous replication
  • • Can be promoted to standalone DB

Use Cases

  • • Read-heavy database workloads
  • • Business reporting and analytics
  • • Disaster recovery scenarios
  • • Cross-region data distribution

Important Notes

  • • Automatic backups must be enabled
  • • Replication lag varies by workload
  • • Each replica has its own endpoint
  • • Can have different instance classes
# Create read replica
aws rds create-db-instance-read-replica \
  --db-instance-identifier mydb-replica \
  --source-db-instance-identifier mydb \
  --db-instance-class db.t3.micro
