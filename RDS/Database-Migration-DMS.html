AWS Database Migration Service (DMS)

AWS DMS helps migrate databases to AWS quickly and securely. It supports homogeneous migrations (same database engine) and heterogeneous migrations (different database engines) with minimal downtime.

Key Benefits: Minimal downtime, continuous data replication, broad database support, cost-effective, and secure migration process.

Homogeneous Migrations

Same Database Engine

  • • Oracle to Amazon RDS for Oracle
  • • MySQL to Amazon RDS for MySQL
  • • PostgreSQL to Amazon RDS for PostgreSQL
  • • SQL Server to Amazon RDS for SQL Server

Migration Process

  • • Schema and data migration
  • • Native database tools can be used
  • • Simpler migration process
  • • Faster migration timeline

Tools & Methods

  • • AWS DMS for continuous replication
  • • Native backup/restore
  • • Import/export utilities
  • • Snapshot-based migration

Heterogeneous Migrations

Different Database Engines

  • • Oracle to Amazon Aurora PostgreSQL
  • • SQL Server to Amazon Aurora MySQL
  • • MySQL to Amazon Aurora PostgreSQL
  • • Oracle to Amazon Redshift

Migration Process

  • • Schema conversion required
  • • Data type mapping
  • • Code conversion (stored procedures)
  • • Application testing required

Required Tools

  • • AWS Schema Conversion Tool (SCT)
  • • AWS DMS for data migration
  • • Application code modification
  • • Extensive testing and validation

DMS Architecture & Components

Replication Instance

  • • Compute resource: EC2 instance that runs replication tasks
  • • Instance classes: T3, C5, R5, M5 instance types
  • • Multi-AZ: High availability deployment option
  • • VPC deployment: Network isolation and security
  • • Storage: Allocated storage for logs and cached data

Source Endpoint

  • • Database location: On-premises, EC2, or RDS
  • • Connection details: Hostname, port, credentials
  • • SSL configuration: Encrypted connections
  • • Extra connection attributes: Engine-specific settings
  • • Test connection: Validate connectivity

Target Endpoint

  • • AWS services: RDS, Aurora, Redshift, S3
  • • Configuration: Connection parameters and settings
  • • Load settings: Batch size, commit rate
  • • Error handling: Retry logic and error tables
  • • Validation: Data integrity checks

Migration Task Types

Full Load

  • • One-time migration: Complete data transfer
  • • Bulk transfer: All existing data
  • • Downtime required: Source database read-only
  • • Use case: Initial data migration
  • • Performance: Parallel loading for speed

Change Data Capture (CDC)

  • • Ongoing replication: Real-time data sync
  • • Transaction logs: Captures changes from logs
  • • Minimal impact: Low overhead on source
  • • Use case: Continuous synchronization
  • • Latency: Near real-time replication

Full Load + CDC

  • • Complete migration: Initial load + ongoing sync
  • • Minimal downtime: Continuous operation
  • • Cutover process: Switch when synchronized
  • • Use case: Production migrations
  • • Validation: Data consistency checks

AWS Schema Conversion Tool (SCT)

Core Capabilities

Schema Conversion
  • • Database objects: Tables, views, indexes, constraints
  • • Data types: Automatic type mapping and conversion
  • • Stored procedures: Code conversion and optimization
  • • Functions: User-defined functions and triggers
Assessment Reports
  • • Complexity analysis: Migration effort estimation
  • • Compatibility issues: Identify conversion challenges
  • • Manual actions: Items requiring manual intervention
  • • Cost estimation: Target infrastructure sizing

Supported Conversions

Source Databases
  • • Oracle Database
  • • Microsoft SQL Server
  • • MySQL
  • • PostgreSQL
  • • IBM Db2
  • • SAP ASE
Target Databases
  • • Amazon Aurora (MySQL/PostgreSQL)
  • • Amazon RDS (MySQL/PostgreSQL/Oracle/SQL Server)
  • • Amazon Redshift
  • • Amazon DocumentDB
  • • Amazon DynamoDB

Migration Best Practices

Planning Phase

  • • Assess source database complexity
  • • Choose appropriate target engine
  • • Plan network connectivity
  • • Estimate migration timeline
  • • Prepare rollback strategy

Execution Phase

  • • Start with non-production environments
  • • Monitor replication lag and performance
  • • Validate data integrity continuously
  • • Test application functionality
  • • Plan cutover window carefully

Post-Migration

  • • Performance tuning and optimization
  • • Monitor application performance
  • • Update connection strings
  • • Implement backup strategies
  • • Decommission source systems

SAA-C03 Exam Key Points

  • • DMS Components: Replication instance, source/target endpoints, migration tasks
  • • Migration Types: Full load, CDC, Full load + CDC for minimal downtime
  • • SCT: Required for heterogeneous migrations, provides assessment reports
  • • Homogeneous: Same engine migrations, simpler process
  • • Heterogeneous: Different engines, requires SCT + DMS
  • • Multi-AZ: DMS replication instances support high availability
