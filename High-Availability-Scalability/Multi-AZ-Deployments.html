Multi-AZ Deployments

High availability across Availability Zones

Multi-AZ Architecture Overview

What are Availability Zones?

Physical Separation: Distinct data centers within a region

Low Latency: High-speed, low-latency network connections

Fault Isolation: Independent power, cooling, and networking

Multi-AZ Benefits

High Availability: 99.99% uptime SLA achievable

Automatic Failover: Seamless failover mechanisms

Load Distribution: Traffic spread across zones

RDS Multi-AZ Deployments

Synchronous Replication

Primary database synchronously replicates to standby in different AZ.

Features:

  • • Zero data loss (RPO = 0)
  • • Automatic failover
  • • No read access to standby
  • • Same region deployment
RTO: 1-2 minutes for automatic failover

Automatic Failover

AWS automatically handles failover without manual intervention.

Triggers:

  • • Primary DB instance failure
  • • AZ outage
  • • DB instance server type change
  • • OS patching
Process: DNS record updated to standby

Maintenance Benefits

Zero-downtime maintenance and patching operations.

Operations:

  • • OS patching on standby first
  • • Failover to patched standby
  • • Patch original primary
  • • Minimal downtime
Benefit: Maintenance without service interruption

Multi-AZ DB Clusters

Enhanced Multi-AZ with readable standby replicas.

Features:

  • • Up to 2 readable standbys
  • • Read traffic distribution
  • • Faster failover (< 35 seconds)
  • • MySQL and PostgreSQL
Advantage: Read scaling + high availability

Cost Considerations

Understanding the cost implications of Multi-AZ deployments.

Costs:

  • • ~2x instance cost
  • • No additional storage cost
  • • No data transfer charges
  • • Backup from standby (free)
ROI: Cost justified by availability gains

Performance Impact

Minimal performance impact with synchronous replication.

Impact:

  • • Slight write latency increase
  • • No read performance impact
  • • Network bandwidth usage
  • • CPU overhead minimal
Typical: 5-10% write latency increase

EC2 Multi-AZ Deployment Patterns

Load Balancer Distribution

Distribute EC2 instances across multiple AZs with load balancing.

Configuration:

  • • ALB/NLB across multiple AZs
  • • EC2 instances in each AZ
  • • Health checks enabled
  • • Cross-zone load balancing
Benefit: Automatic traffic rerouting on AZ failure

Auto Scaling Groups

ASG automatically maintains capacity across multiple AZs.

Features:

  • • Even distribution across AZs
  • • Automatic replacement
  • • AZ rebalancing
  • • Health check integration
Resilience: Survives single AZ failure

EBS Multi-AZ Strategy

EBS volumes are AZ-specific, requiring backup and restore strategies.

Strategies:

  • • Regular EBS snapshots
  • • Cross-AZ snapshot copying
  • • Automated backup policies
  • • Fast Snapshot Restore
RTO: Depends on volume size and FSR

Features:

Lambda

Features:

  • • Built-in fault tolerance
  • • No configuration needed

Disaster Recovery Strategies

RTO vs RPO

RTO (Recovery Time Objective): Maximum acceptable downtime

RPO (Recovery Point Objective): Maximum acceptable data loss

Trade-off: Lower RTO/RPO = higher cost

DR Patterns

Backup & Restore: Lowest cost, highest RTO

Pilot Light: Core systems ready, scale on demand

Warm Standby: Scaled-down version running

Multi-Site Active/Active: Lowest RTO, highest cost

Multi-AZ Best Practices

Design Principles

Assume failure: Design for component and AZ failures

Loose coupling: Minimize dependencies between AZs

Stateless design: Store state externally when possible

Implementation Best Practices

Health checks: Implement comprehensive health monitoring

Chaos testing: Regularly test failure scenarios

Monitoring: Track availability and performance metrics
