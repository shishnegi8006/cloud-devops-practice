Auto Scaling Groups

Dynamic scaling and automated capacity management

Auto Scaling Overview

What is Auto Scaling?

Dynamic Capacity: Automatically adjusts EC2 instance count

Cost Optimization: Scale down during low demand periods

High Availability: Maintains desired capacity across AZs

Key Components

Launch Template: Defines instance configuration

Scaling Policies: Rules for when to scale in/out

Health Checks: Monitors instance health status

Launch Templates & Configurations

Launch Templates

Modern approach for defining instance launch parameters with versioning.

Features:

  • • Version control and rollback
  • • Mixed instance types support
  • • Spot instance integration
  • • Advanced networking options
Recommended: Use instead of launch configurations

Instance Configuration

Define AMI, instance type, security groups, and storage settings.

Parameters:

  • • AMI ID and instance type
  • • Key pair and security groups
  • • IAM instance profile
  • • User data scripts
Best Practice: Use golden AMIs for consistency

Mixed Instance Types

Use multiple instance types and purchase options for cost optimization.

Options:

  • • On-Demand and Spot instances
  • • Multiple instance families
  • • Allocation strategies
  • • Capacity rebalancing
Benefit: Up to 90% cost savings with Spot

Scaling Policies

Target Tracking

Maintains a specific metric value by automatically scaling capacity.

Metrics:

  • • Average CPU utilization
  • • Request count per target
  • • Network bytes in/out
  • • Custom CloudWatch metrics

Example:

Maintain 70% CPU utilization

Recommended: Easiest to configure and manage

Step Scaling

Scales based on the size of the alarm breach with multiple steps.

Configuration:

  • • Multiple scaling adjustments
  • • Metric value ranges
  • • Cooldown periods
  • • CloudWatch alarms

Example:

+1 instance if CPU 70-80%, +2 if >80%

Use Case: Fine-grained scaling control

Simple Scaling

Single scaling adjustment based on CloudWatch alarm state.

Features:

  • • Single scaling action
  • • Cooldown period required
  • • CloudWatch alarm trigger
  • • Simple configuration

Limitation:

Cannot scale during cooldown period

Status: Legacy, use target tracking instead

Scheduled Scaling

Proactive scaling based on predictable time patterns.

Schedule Types:

  • • One-time scheduled actions
  • • Recurring schedules
  • • Cron expressions
  • • Time zone support

Example:

Scale up at 8 AM, scale down at 6 PM

Best For: Predictable traffic patterns

Predictive Scaling

Uses machine learning to forecast traffic and scale proactively.

Features:

  • • ML-based forecasting
  • • Historical data analysis
  • • Proactive capacity provisioning
  • • Forecast-only or scale mode

Requirement:

Minimum 14 days of historical data

Benefit: Reduces scaling lag time

Warm Pools

Pre-initialized instances for faster scaling response times.

States:

  • • Stopped (EBS-backed only)
  • • Running (immediate availability)
  • • Hibernated (faster than stopped)

Benefit:

Reduces instance launch time

Cost: Pay for EBS storage only when stopped

Health Checks & Monitoring

EC2 Health Checks

Basic instance health monitoring using EC2 status checks.

Checks:

  • • System status checks
  • • Instance status checks
  • • Instance reachability
Default: Enabled by default for all ASGs

ELB Health Checks

Application-level health monitoring through load balancer checks.

Features:

  • • HTTP/HTTPS endpoint checks
  • • Custom health check paths
  • • Configurable thresholds
  • • More sensitive than EC2 checks
Recommended: Use with load balancers

Health Check Grace Period

Time to allow new instances to initialize before health checks begin.

Configuration:

  • • Default: 300 seconds
  • • Adjustable based on app startup time
  • • Prevents premature termination
Best Practice: Set based on application startup time

Lifecycle Hooks

Launch Lifecycle Hooks

Pause instance launch to perform custom initialization tasks.

Use Cases:

  • • Software installation and configuration
  • • Log file copying
  • • Service registration
  • • Custom health checks

Integration:

  • • SNS notifications
  • • SQS messages
  • • Lambda functions

Terminate Lifecycle Hooks

Pause instance termination to perform cleanup tasks.

Use Cases:

  • • Data backup and archival
  • • Log file extraction
  • • Service deregistration
  • • Connection draining

Timeout:

Default 1 hour, configurable up to 48 hours

Auto Scaling Best Practices

Configuration Best Practices

Launch templates: Use instead of launch configurations

Multiple AZs: Distribute instances across availability zones

Health checks: Enable ELB health checks for web applications

Scaling Best Practices

Target tracking: Preferred scaling policy for most use cases

Cooldown periods: Prevent rapid scaling oscillations

Monitoring: Use CloudWatch metrics and alarms
