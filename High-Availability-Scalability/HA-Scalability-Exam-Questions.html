Load Balancer Selection

A gaming company needs to load balance TCP traffic for their multiplayer game servers. They require ultra-low latency, static IP addresses, and the ability to preserve the source IP of clients. Which load balancer should they choose?

A) Application Load Balancer (ALB)
B) Network Load Balancer (NLB)
C) Gateway Load Balancer (GWLB)
D) Classic Load Balancer (CLB)

Explanation:

Correct Answer: B
Network Load Balancer (NLB) operates at Layer 4 (TCP/UDP) and provides ultra-low latency, static IP addresses, and preserves the source IP of clients. It's specifically designed for high-performance applications like gaming servers that require these features. ALB operates at Layer 7 and adds latency, GWLB is for network appliances, and CLB is legacy.

Auto Scaling Policy

An e-commerce application experiences predictable traffic patterns with high load during business hours (9 AM - 6 PM) and low load at night. The application also needs to handle unexpected traffic spikes. What's the best Auto Scaling strategy?

A) Use only dynamic scaling with target tracking policy
B) Combine scheduled scaling for predictable patterns with target tracking for unexpected spikes
C) Use only scheduled scaling actions
D) Use simple scaling policies only

Explanation:

Correct Answer: B
The best approach combines scheduled scaling (proactive scaling for known patterns like business hours) with target tracking scaling (reactive scaling for unexpected spikes). Scheduled scaling ensures capacity is ready before the predictable load, while target tracking handles unexpected traffic variations. This provides both cost optimization and performance assurance.

3

Multi-AZ High Availability

A company wants to ensure their web application can survive the failure of an entire Availability Zone. They're using ALB, Auto Scaling Groups, and RDS. What configuration ensures the highest availability?

A) Deploy everything in a single AZ with multiple instances
B) Deploy ALB across multiple AZs, ASG with instances in multiple AZs, and RDS Multi-AZ
C) Deploy ALB in one AZ, ASG across multiple AZs, RDS in single AZ
D) Use multiple regions instead of multiple AZs

Explanation:

Correct Answer: B
For AZ-level fault tolerance, all components must be deployed across multiple AZs: ALB automatically distributes across AZs in the region, ASG should have instances in multiple AZs with appropriate subnet configuration, and RDS Multi-AZ provides automatic failover to a standby in another AZ. This ensures the application continues operating even if an entire AZ fails.

4

Health Check Configuration

An Auto Scaling Group is configured with both ELB health checks and EC2 health checks. An instance fails the ELB health check but passes the EC2 health check. What happens to the instance?

A) The instance remains healthy since EC2 health check passed
B) The instance is marked unhealthy and will be terminated and replaced
C) The instance is temporarily removed from the load balancer but not terminated
D) Only the ELB stops sending traffic, ASG takes no action

Explanation:

Correct Answer: B
When both ELB and EC2 health checks are enabled, ASG considers an instance unhealthy if it fails either health check. If an instance fails the ELB health check (even if EC2 health check passes), ASG will mark it as unhealthy, terminate it, and launch a replacement. This ensures that only instances capable of serving traffic remain in the ASG.

5

Disaster Recovery Strategy

A financial services company needs a disaster recovery solution with RTO (Recovery Time Objective) of 1 hour and RPO (Recovery Point Objective) of 15 minutes. They want to minimize costs while meeting these requirements. What's the best DR strategy?

A) Hot standby with full capacity in secondary region
B) Warm standby with minimal capacity and RDS Cross-Region Read Replicas
C) Cold standby with automated infrastructure deployment
D) Backup and restore from S3

Explanation:

Correct Answer: B
Warm standby provides the right balance for 1-hour RTO and 15-minute RPO requirements while minimizing costs. It maintains minimal infrastructure in the secondary region (reducing costs compared to hot standby) but can scale up quickly. RDS Cross-Region Read Replicas provide near real-time data replication (meeting 15-minute RPO) and can be promoted to primary within the 1-hour RTO requirement.
